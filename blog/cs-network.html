CS 기초 - Network

Authors
  • Name
    Hoehyeon Jung
    Twitter

Intro

개발을 진행하다보면 항상 사용하기 때문에 간과하기 쉬운 네트워크에 대해 다시 한번 알아보고자 한다. 이를 가장 일반적인 상황으로부터 네트워크에 대해 설명하고자 한다.

Situation

브라우저에 URL을 입력하면 어떤 일이 일어날까?

  1. 사용자는 브라우저에 URL을 입력한다.
  2. 브라우저는 해당 URL을 DNS 서버에 요청한다.
  3. DNS 서버는 해당 URL에 대한 IP 주소를 응답한다.
  4. 브라우저는 해당 IP 주소로 HTTP 요청을 보낸다.

각 과정에 대해 상세하게 살펴보자.

1. 사용자는 브라우저에 URL을 입력한다.

사용자가 브라우저에 URL을 입력하면 브라우저는 해당 URL을 해석해야 한다. 기본적으로 서버는 특정 공인 IP 상에서 운영되고 있고, 이를 통해 서버에 접근할 수 있다. 하지만 이러한 무작위적인 숫자들은 IP는 사용자들이 외우기 어렵기 때문에 의미있는 문자인 도메인을 사용하게 된다. 이러한 도메인을 IP로 변환해주는 역할을 하는 것이 DNS 서버이다.

2. 브라우저는 해당 URL을 DNS 서버에 요청한다.

DNS 서버는 해당 URL에 대한 IP 주소를 제공하는 역할을 가지고 있다. 이를 위해 https://gonewbie.github.io/blog/cs-network/ URL을 분리하면 아래와 같이 이루어져 있다.

  • 통신 규약(Protocol)

흔히 URL 앞에 https:// 혹은 http:// 로 시작되는 도메인이 많다. 이러한 부분을 통신 규약으로 주어지며 브라우저는 이를 해석하여 해당 규약에 맞는 서버에 요청을 보낸다. 일반적으로 HTTPS는 443 포트, HTTP는 80 포트를 사용한다.

  • 도메인(Domain)

일반적인 서버 고유의 이름에 대응되는 부분이 gonewbie.github.io 이다. 이러한 도메인을 통해 서버에 접근할 수 있다.

  • 경로(Path)

도메인 이후에 오는 /blog 부분이다. 서버에서 정적/동적 자원 제공을 위한 경로 해당 경로를 지정하여 서버 자원을 관리한다.

  • 자원(Resource)

경로 이후에 오는 /cs-network 부분이다. 해당 자원을 통해 서버의 특정 자원에 접근할 수 있다.

이 중에서 도메인 부분을 DNS 서버에 전달하게 된다.

3. DNS 서버는 해당 URL에 대한 IP 주소를 응답한다.

DNS 서버에 접근하고 IP 주소를 가져오는 것은 복잡하기도 하고 상당한 시간을 소요하기 때문에 기본적으로 캐싱을 통해 이를 해결한다. 이를 위해 브라우저는 먼저 자신의 캐시를 확인하고, 없을 경우 운영 체제, 라우터 로컬 캐시, 인터넷 서비스 제공업체(ISP)의 DNS 재귀 확인자(recursive resolver)에 요청을 보내게 된다. 만일 ISP의 DNS 재귀 확인자 여기에 저장되지 않았다면 루트 DNS 서버에 요청을 보내게 된다.

(이미지 출처: https://www.cloudflare.com/img/learning/dns/dns-server-types)

모든 DNS 재귀 확인자에 알려져 있는 루트 DNS 서버는 gonewbie.github.io 도메인의 .io 도메인 TLD(Top Level Domain) 서버에 요청을 보내게 된다. TLD 서버는 해당 도메인의 권한있는 네임 서버(authoritative nameserver)를 알려주게 되고, 이를 통해 해당 도메인의 네임 서버에 요청을 보내게 된다. 권한있는 네임 서버는 해당 도메인의 IP 주소를 DNS A 레코드(IPv4)에서 조회하여 알려주게 된다.

4. 브라우저는 해당 IP 주소로 HTTP 요청을 보낸다.

이제 브라우저는 해당 도메인을 가진 서버의 IP를 알게 되었다. 이를 통해 서버에 HTTP 요청을 보내게 된다. 이를 위해 브라우저는 해당 IP 주소로 TCP 연결을 시도하게 된다. 일반적으로 모든 요청에 대해 서버와 직접 통신하는 것은 지리상 효율적이지 않다. 따라서 정적인 파일 등은 CDN 등의 서비스를 통해 사용자와 지리적으로 가까운 위치에서 문서들을 빠르게 제공한다. 이후 IP 주소로 해당 서버와 직접 혹은 로드 밸런서를 경우하여 서버에 연결된다.

4-1. HTTP 연결

HTTP의 경우 기본적으로 TCP/IP를 사용한다. TCP는 연결 지향적이기 때문에 연결을 위한 3-way handshake, 연결 종료를 위한 4-way handshake를 통해 연결을 맺게 된다. 하지만 HTTP 버전에 따라 연결을 유지하는 방식이 다르다. HTTP 1.0의 경우 연결을 유지하지 않지만, HTTP/1.1의 경우 기본적으로 연결을 유지하게 된다. 이를 통해 요청을 보낼 때마다 연결을 맺고 끊는 것이 아니라 연결을 유지한 상태에서 요청을 보내게 된다. 또한 데이터를 Pipelining 하여 여러 개의 데이터를 보내게 된다. 여기에 HTTP/2가 되면서, Header를 압축하여 전송하고, HTTP/1.1의 한계였던 Head of Line Blocking을 해결하였다. (순차적으로 데이터를 전송해야 하는 문제를 해결 frame의 interleaving을 통해 해결) Stream 단위로 데이터를 전송하게 되어, 여러 개의 데이터를 동시에 전송할 수 있게 되었다. 또한 리소스 인라인화로 무거운 리소스를 서버 푸시를 통해 비동기적으로 전송할 수 있게 되었다.

4-2. HTTPS 연결

HTTPS의 경우 기본적으로 TCP/IP를 사용한다. 하지만 HTTP와 다르게 TLS를 통해 암호화된 연결을 맺게 된다. TLS는 기본적으로 공개키 암호화 방식을 사용한다. 이를 통해 서버는 공개키를 통해 암호화된 데이터를 받을 수 있고, 클라이언트는 개인키를 통해 데이터를 복호화할 수 있다. 이를 통해 데이터를 안전하게 전송할 수 있다.
