몽고 디비 쿼리 작성

특정 상품의 구매 기록에서 간단한 통계 결과를 출력하여 사용자에게 제공하는 기능을 담당 API를 작성하는 업무가 주어졌다.
이를 위해서 몽고 디비 쿼리 작성을 위해 겪었던 시행착오나 쿼리 개선 방안 방법 등에 대한 기록을 남기고자 해당 포스트를 작성하기로 했다.

조회하는 Collection 내에서 연산되어서 출력하도록 나오도록 하기 위해서 aggregation을 사용하였다.

MongoDB의 aggregation은 기본적으로 multi-stage pipeline을 수행하기 때문에 각 stage 연산을 아래와 같은 배열로 받아들이게 된다.

db.collection.aggregate{[ { stage }, { stage }, … ])

이러한 스테이지 별 연산을 순차적으로 수행하기 때문에 순서를 잘 정하여 연산하는 것이 중요하다고 볼 수 있다.
기본적으로 데이터베이스에서 데이터를 필터링하고, 필터링된 데이터를 기반으로 연산하는 과정을 거치는 것으로 볼 수 있다.

가상의 비즈니스 로직 구성

이러한 aggregate에 대한 항목에 대한 예제를 다뤄보기 위해서 우리가 다룰 Mongo DB 샘플 데이터는 Cloud 환경으로 제공해주는 mongoDB atlas라는 사이트가 있다.
개발자라면 혹할 만한 조건들이 가득한데 특정 조건들만 선택하면 무료로 DB 인스턴스를 제공해준다.
클라우드 환경에서 DB가 접근 가능하다보니 개발자는 개발 환경마다 별도로 DB를 구성하는 수고로움을 덜 수 있다.
MongoDB Atlas 클라우드 서버 구성에 대한 내용은 본 post 주제 범위를 넘어가므로 생략하도록 하겠다.
이러한 MongoDB Atlas에서 제공해주는 예제 DB를 활용하여 MongoDB aggregation Pipeline을 이해해보자.

sample DB로는 sample-supplies를 사용할 것이다.
가상의 홈쇼핑 페이지에서 당신에게 주어진 상황을 다음과 같다고 가정해보자. 연말연시를 맞아 사무용품을 구매한 특정 사용자 층이 쿠폰을 사용한 결제 내역을 추적하고자 한다.
관리자 페이지에서 이러한 목록을 조회할 수 있다는 배경이 주어진다면, 목록을 추스리는 비즈니스 로직만 수행하면 되므로 이에 대한 과정은 다음과 같은 단계로 나뉠 수 있다.

쿠폰을 사용하여 물품을 구매한 남성들의 구매 경로 및 금액을 추리는 방법

  • 고객 정보 중에 쿠폰을 사용하여 물품을 구매한 남성들의 목록을 추린다.
  • 사무용품을 구매한 해당 사용자들의 구매 물품들의 총 합을 구한다.

filtering

aggregate에 있어서 가장 기본적인 필터링은 $match 라고 볼 수 있다.
기본적이지만 강력한 연산자인 $match는 각 document 내부 항목들에 대한 필터링을 거쳐서 데이터베이스 조회 결과물들을 한 번 걸러낸다.
Query 연산자를 $match 스테이지에서 그대로 사용할 수 있기 때문에 기존 mongoDB 사용자라면 기존과 동일하게 사용가능하다.

쿼리를 작성하는 데 있어, mongodb Client로는 Robo3t를 사용하였다. 따라서 해당 쿼리의 출력 결과 등이 보여진 예시와 다를 수도 있는 점이 있을 수도 있다.

주어진 조건에 맞는 쿼리를 작성해보면 다음과 같을 것이다.

  • 고객 정보 중에 쿠폰을 사용한 남성들의 목록을 추린다.

$match stage 후 결과 예시

컬렉션 내 5000개의 데이터에서 남성이고 쿠폰을 사용한 사용자를 1차적으로 추리게 된다.
또한 데이터를 필터링 하는데 있어 기존의 find등 메소드에서 사용하는 Query 연산자 등 다양한 연산자 역시 각 파이프라인마다 활용이 가능하므로
이를 적절하게 활용하면 효과적으로 데이터를 필터링할 수 있다.

이러한 연산자 항목들에 대해 자세하게 알고 싶다면 mongoDB Document를 참고하자.

grouping

보통 통계를 위한 데이터 처리에 있어서 필요한 것은 같은 분류의 데이터끼리 합치는 것이다. 다행히도 mongoDB에서는 이를 위해서 $group이라는 멋진 연산자가 있다.
$group 연산자는 항목의 이름을 정의하고 이를 위한 그룹의 세부 조건에 따라서 그룹화한 항목들을 가져온다.

쿼리는 위에서 사용하던 aggregate 쿼리에 이어서 사용한다고 보면 다음과 같을 것이다.

  • 해당 사용자들의 구매 물품들의 총 합을 구한다.

쿼리 결과 예시

$group 스테이지에서는 이전 스테이지의 결과물들로부터 새로운 결과물을 만드는 작업을 진행한다.
새로운 document를 생성하는만큼, 고유한 _id를 필수적으로 지정해줘야 한다.
위의 예제에서는 구매 방식에 대해서 고유값으로 정하고, 사무용품을 같이 구매했던 사용자들에 한하여 숫자를 더하는 과정을 거치게 되었다.
[explain을 통해 $match에서 필터링 / $switch 분기에서 필터링 사진 비교]
(bug?: match랑 switch랑 비교시 계산이 미묘하게 다르다. edge case가 있는지 살펴보자..)

즉, totalPrice를 구하는데에 있어서 이전 스테이지에서 필터링을 거쳤다고 해서 이후 스테이지에서 필터링을 하지 못하는 것이 아니라
이후 스테이지에서도 독립적으로 필터링과 같은 연산을 다시 활용할 수 있는 것이다.
위의 예제의 경우, MongoDB 3.4 버젼부터 지원하는 $switch 연산자를 사용하였다.
$switch 연산자는 이후에 나오는 branch 조건에 따라서 연산하도록 처리하였는데 위와 같은 경우 $sum 연산을 수행할 때,
각 document 구매 항목들 중에서 office tag를 포함한 구매 내역만 덧셈에 포함하도록 분기를 거친 것이다.

일반 프로그래밍 언어에서의 switch 구문처럼 확실하게 구분된 조건에서만 사용하는 것을 모델로 삼은 것인지, $elemMatch와 같은 내부 항목을 조회하여 비교하는 연산자를 branch로 활용할 수 없다.
다만, $switch 연산에서 각 항목이 object로 구성되어 있어 내부 property를 비교하는 것을 switch로 분기하여 연산하기 보다는
해당 항목을 $match와 같은 필터링 스테이지에서 처리하거나, 불가피한 경우 내부 property를 조회하는 복잡한 분기를 활용하지 않도록 하는 것이 바람직할 것 같다.

($switch를 활용한 논리적 분기가 포함된 코드 넣기)

$facet

또한 aggregation 관련 연산자로 $facet 연산자가 존재한다.
$facet 연산자 역시 MongoDB 3.4버젼에서 추가된 버젼으로 본래 특정 group 내에서 여러 개의 연산자를 처리해주는 연산자이다.
즉, 하나의 스테이지 안에 별도의 스테이지들이 존재하는 것이다.

쿼리 결과

쿼리 결과를 살펴보면 $match, $group의 경우 동일한 결과를 분류로 나누어 계산하는 반면, $facet의 경우 해당 스테이지 내에서 각각의 그룹별로 별도의 연산을 수행하는 것을 알 수 있다.
따라서 그룹을 분류하여 연산하는 작업의 경우 filtering({$match:…}) -> grouping({$group:…})과 같은 연산을 하고,
별도의 성격을 가진 연산 결과물들을 묶어서 출력하고자 하는 경우 $facet을 활용하는 것이 적절할 것 같다.

$mongoDB atlas 주의 사항

filtering => grouping 과 같은 연산을 javascript 코드로 구현하는 mapreduce 등이 제한되는 문제가 있는 등 일부 연산자나 명령어에 대해서 제한이 있다.

참고) MongoDB atlas 제한

indexing

쿼리를 작성하는 것은 1회성에 그칠 것이 아니라 향후 서비스가 성장하고 데이터가 축적됨에 따라 조회 시간 역시 같이 증가하는 문제를 가지고 있기 때문에 끊임없이 고민하면서 설계하고 쿼리 시간을 고려해봐야 한다.
따라서 이러한 쿼리 검색을 도와주는 강력한 도구로는 indexing이 있다.
특정 쿼리에 도움이 되는 index를 사전에 생성함으로 인해 query 실행 시 기존보다 훨씬 빠른 순회가 가능하다.
