Mongodb Aggregate

나는 MongoDB를 얼마나 잘 알고 쓰고 있는가? 이 물음에 대한 대답은 실무에서 MongoDB를 본격적으로 다루고 본인에게 주어진 업무를 수행하면서 다시 고민해보는 계기가 되었다.
본인에게 주어졌던 업무는 특정 상품의 구매 기록에서 간단한 통계 결과를 출력하여 사용자에게 제공하는 기능을 담당하는 API를 작성하는 업무가 주어졌었다.
이 업무를 수행하기 위해 비즈니스 로직을 작성하는 것 이외에도 DB에서 데이터를 연산하여 fetch하는 방법을 찾아보니 mongoDB는 이를 위한 aggregate framework가 이미 존재했다.
이 포스트는, 이러한 aggregate를 이해해 나가면서 겪었던 시행착오나 쿼리 개선 방안 방법 등에 대한 기록이다.

Aggregate의 구성

Aggregate Framework은 기본적으로 multi-stage pipeline을 수행하기 때문에 각 stage 연산을 아래와 같은 배열로 받아들이게 된다.

db.collection.aggregate{[ { stage }, { stage }, … ])

이러한 스테이지 별 연산을 순차적으로 수행하기 때문에 순서를 잘 정하여 연산하는 것이 중요하다고 볼 수 있다.
기본적으로 데이터베이스에서 데이터를 필터링하고, 필터링된 데이터를 기반으로 연산하는 과정을 거치는 것으로 볼 수 있다.

가상의 비즈니스 로직 구성

이러한 aggregate에 대한 항목에 대한 예제를 다뤄보기 위해서 우리가 다룰 Mongo DB 샘플 데이터는 Cloud 환경으로 제공해주는 mongoDB atlas라는 사이트가 있다.
개발자라면 혹할 만한 조건들이 가득한데 특정 조건들만 선택하면 무료로 DB 인스턴스를 제공해준다.
클라우드 환경에서 DB가 접근 가능하다보니 개발자는 개발 환경마다 별도로 DB를 구성하는 수고로움을 덜 수 있다.
MongoDB Atlas 클라우드 서버 구성에 대한 내용은 본 post 주제 범위를 넘어가므로 생략하도록 하겠다.
이러한 MongoDB Atlas에서 제공해주는 예제 DB를 활용하여 MongoDB aggregation Pipeline을 이해해보자.

Filtering

aggregate에 있어서 가장 기본적인 필터링은 $match 라고 볼 수 있다.
기본적이지만 강력한 연산자인 $match는 각 document 내부 항목들에 대한 필터링을 거쳐서 데이터베이스 조회 결과물들을 한 번 걸러낸다.
1차적으로 데이터를 걸러낸 이후에 연산하는 것이 효율적인 쿼리를 수행할 수 있기 때문에 대체로 aggregation stage의 맨 처음에 위치하게 된다.
mongoDB에 접속할 Client로는 Robo3t를 선택하였다.

쿼리를 작성하는 데 있어, mongodb Client로는 Robo3t를 사용하였다. 따라서 해당 쿼리의 출력 결과 등이 보여진 예시와 다를 수도 있는 점이 있을 수도 있다.

주어진 조건에 맞는 쿼리를 작성해보면 다음과 같을 것이다.

  • 고객 정보 중에 쿠폰을 사용하여 물품을 구매한 남성들의 목록을 추린다.

또한 데이터를 필터링 하는데 있어 기존의 find등 메소드에서 사용하는 Query 연산자 등 다양한 연산자 역시 각 파이프라인마다 활용이 가능하므로 이를 적절하게 활용하면 효과적으로 데이터를 필터링할 수 있다.

Grouping

보통 통계를 위한 데이터 처리에 있어서 필요한 것은 같은 분류의 데이터끼리 합치는 것이다. 다행히도 mongoDB 에는 이를 위해서 $group이라는 멋진 연산자가 있다.
$group 연산자는 항목의 이름을 정의하고 이를 위한 그룹의 세부 조건에 따라서 그룹화한 항목들을 가져온다.

쿼리는 위에서 사용하던 aggregate 쿼리에 이어서 사용한다고 보면 다음과 같을 것이다.

  • 사무용품을 구매했던 해당 사용자들의 구매 물품들만의 총 합을 구한다.

쿼리 결과 예시

$group 스테이지에서는 이전 스테이지의 결과물들로부터 새로운 결과물을 만드는 작업을 진행한다.
새로운 document를 생성하는만큼, 고유한 _id를 필수적으로 지정해줘야 한다.
위의 예제에서는 구매 방식에 대해서 고유값으로 정하고, 사무용품을 구매한 이력이 있는 사용자들에 한하여 숫자를 더하는 과정을 거치게 되었다.

즉, totalPrice를 구하는데에 있어서 이전 스테이지에서 필터링을 거쳤다고 해서 이후 스테이지에서 필터링을 하지 못하는 것이 아니라 이후 스테이지에서도 독립적으로 필터링과 같은 연산을 다시 활용할 수 있는 것이다.
위의 예제의 경우, MongoDB 3.4 버젼부터 지원하는 $switch 연산자를 사용하였다.
$switch 연산자는 이후에 나오는 branch 조건에 따라서 연산하도록 처리하였는데 위와 같은 경우 $sum 연산을 수행할 때, 각 document 구매 항목들 중에서 office tag를 포함한 구매 내역만 덧셈에 포함하도록 분기를 거친 것이다.

일반 프로그래밍 언어에서의 switch 구문처럼 확실하게 구분된 조건에서만 사용하는 것을 모델로 삼은 것인지, $elemMatch와 같은 내부 항목을 조회하여 비교하는 등의 비교적 복잡한 연산자를 case로 활용할 수 없다.
다만, $switch 연산에서 각 항목이 object로 구성되어 있어 내부 property를 비교하는 것을 switch로 분기하여 연산하기 보다는 해당 항목을 $match와 같은 필터링 스테이지에서 처리하거나, 불가피한 경우 내부 property를 조회하는 복잡한 분기를 활용하지 않도록 하는 것이 바람직할 것 같다.

$facet

또한 aggregation 관련 연산자로 $facet 연산자가 존재한다.
$facet 연산자 역시 MongoDB 3.4버젼에서 추가된 연산자로 특정 stage 내에서 여러 개의 연산자를 처리해주는 연산자이다.
즉, 하나의 스테이지 안에 별도의 스테이지들이 존재하는 것이다.

  • 구매 물품들에 대해서 남성 사용자에 대한 아이템 이름과 항목만을 추린다.
  • 연령별 분류를 위해 20대씩 묶어서 이메일 목록을 가져온다.

쿼리 결과

쿼리 결과를 살펴보면 filtering($match), grouping($group) 과정을 활용한 경우 동일한 연산 과정에 대해서 연산을 진행하고 결과를 분류하여 출력해주는 반면, $facet의 경우 해당 스테이지 내에서 각각의 그룹별로 별도의 연산을 수행하는 것을 알 수 있다.
따라서 그룹을 분류하여 연산하는 작업의 경우 filtering({$match:…}) -> grouping({$group:…})과 같은 연산을 하고, 별도의 성격을 가진 연산 결과물들을 묶어서 출력하고자 하는 경우 $facet을 활용하는 것이 적절할 것 같다.

참고) $bucket = $group + $switch + $sort

$bucket 연산자는 mongoDB 3.4 버전에 추가된 것으로, boundary라는 경계 구간 사이의 값들 기준으로 group을 묶어서 연산을 하는 연산자이다.
어떻게 보면 위의 연산에서 $group 연산자와 $switch 연산자를 묶어서 사용한 것으로 볼 수 있는데 실제로 explain()을 실행해보면 이와 동일함을 확인할 수 있다.

explain 결과를 보면 $bucket 연산자는 정확히는 $group, $switch, $sort를 사용하는 것을 확인할 수 있다.

Indexing

쿼리를 작성하는 것은 1회성에 그칠 것이 아니라 향후 서비스가 성장하고 데이터가 축적됨에 따라 조회 시간 역시 같이 증가하는 문제를 가지고 있기 때문에 끊임없이 고민하면서 설계하고 쿼리 시간을 고려해봐야 한다.
따라서 이러한 쿼리 검색을 도와주는 강력한 도구로는 indexing이 있다.
특정 쿼리에 도움이 되는 index를 사전에 생성함으로 인해 query 실행 시 기존보다 훨씬 빠른 순회가 가능하다.

인덱스는 기본적으로 단일 인덱스(single field index), 복합 인덱스(compound index)로 나뉘어진다.

이러한 구분은 인덱스에 사용되는 key 값이 하나인 경우를 단일, 그 이상의 경우를 복합으로 나뉘어 판단하게 된다.
그렇다면 굳이 단일 인덱스를 여러 번 참조하는 게 복합 인덱스를 사용하는 것보다 이점이 없을 것 처럼 보일 수도 있다.
하지만 인덱스가 늘어남에 따라서 해당 Collection에 document가 추가될 떄, 인덱스 늘어난 인덱스 규칙에 맞춰 생성해야 하기 때문에 쓰기에 시간이 소요되게 된다.
따라서, 읽기와 쓰기의 빈도를 조절하여 index를 필요한만큼 생성하는 것이 핵심이다.

이러한 내용을 예제를 통해 간단하게 이해해보자.
예제는 mongoDB Index 메뉴얼을 차용하여 진행하도록 하겠다.

단일 필드 인덱스의 경우, 유일한 필드에 대한 인덱싱을 의미한다.
하지만 일반적으로 쿼리를 진행할 때, 다중 필드를 활용하는 쿼리를 쓸 때는 복합 인덱스가 더 효율적일 것이다.

위의 예제와 같은 복합 인덱싱을 적용한 Collection이 있다 가정해보자.
복합 인덱스에서는 생성된 인덱스 필드들의 순서가 매우 중요하다.
만일 이러한 순서를 지키지 않는다면 인덱스의 효능을 충분히 활용하지 못하거나 심할 경우 전혀 활용하지 못하는 경우가 발생할 수 있다.
userId를 기준으로 오름차순, score를 기준으로 내림차순으로 생성했기 때문에 쿼리를 작성할 때도 이를 염두해서 작성해야 한다.
{userId: 1, score: -1} 인덱스를 기준으로 이와 동일한 query는 물론이고, 이와는 정반대 순서인 {userId: -1, score: 1}의 query field를 조회한다면 해당 인덱스를 충분히 활용하여 쿼리를 진행할 것이다.

Index Prefix

또한 복합 인덱스의 경우에는 이러한 복합 인덱스들의 일부분만을 조회하여도 해당 인덱스를 충분히 활용한 쿼리가 가능하다.
{userId: 1, score: -1} 인덱스의 경우 이러한 부분 인덱스 key를 userId로 볼 수 있는데, 이러한 부분 인덱스들을 index prefix라고 부른다.
이러한 index prefix를 이용하면 하나의 복합 인덱스로 해당 index prefix들까지 다룰 수 있는 효율이 생기므로 이러한 복합 인덱스의 작성을 좀 더 고려해봐야 한다.

지금까지 설명을 기초로 가상의 인덱스를 지정해보고 해당 인덱스에 대한 예제 인덱스들의 성능을 한눈에 가늠해보자.

3.FullScan 진행

이러한 인덱스의 기본을 이해한 채 위에서 사용했던 쿼리들을 최적화 해보자.
우선 index를 사용하는 쿼리는 현재 예제 쿼리 중에서, find/match 등의 쿼리에서는 사용하는 것으로 알려져 있다.
그럼 우선 Filtering => Grouping 부분에 해당되는 aggregate에 대해 index를 생성해보자.

$match부분을 다시 살펴보면, 'customer.gender'와 couponUsed 필드를 기준으로 필터링을 진행했다. 따라서 이 두 항목에 대한 compound Index를 생성해보자.

$ db.getCollection(‘sales’).createIndex({‘customer.gender’: 1, couponUsed: 1})

index가 생성이 되면 이제 위의 쿼리에 대해 index를 통과하는 지 여부를 확인할 수 있는 explain() 메소드를 사용하자.

aggregate()의 경우 .explain()을 앞에 붙여야 실행 가능하다.

explain 결과를 살펴보면, 우선은 aggregate stage를 분류해서 작업이 진행되는 것을 알 수 있다.
$match stage의 경우 index를 활용하기 때문에 queryPlanner 항목을 살펴보면 어떤 인덱스를 실행하는지 자세하게 알 수 있다.
그 중에서 winningPlan을 살펴보면 stage가 FETCH 된 것을 알 수 있는데 이는 index를 활용하여 검색을 진행한 것을 알 수 있다.
이에 대한 상세한 내용은 “inputStage”(혹은 “inputStages”) 항목에서 자세하게 나오는데,
여기서 indexName을 살펴보면, 우리가 생성했던 {'customer.gender': 1, couponUsed: 1} 항목에 대한 기본 인덱스명을 사용한 것을 보고 인덱스의 사용 여부를 알 수 있다.
현재는 데이터가 많이 않아 FULL SCAN과의 차이가 급격하게 발생하지는 않지만, 이러한 차이는 데이터가 커질수록 기하급수적으로 문제가 발생할 수 있으므로 데이터를 분석하여 사전에 인덱스를 설정하는 것이 바람직하다.

또한 switch 구문을 다시 살펴보면, case가 단일한 경우에 대해서는 branches가 필요하지 않으므로 이 부분에 대해서는 index를 활용 가능한 $match로 올리는 것이 바람직하다.
index로 ‘item.tags’를 활용 가능하기 때문에 이에 대한 인덱스를 생성하는 것이 좋다.

db.getCollection(‘sales’).createIndex({‘customer.gender’: 1, couponUsed: 1, ‘items.tags’: 1});

하지만 이렇게 ‘items.tags’ field를 추가하게 되면 기존의 {‘customer.gender’: 1, couponUsed: 1} 인덱스와 겹치게 되는데 기존의 인덱스는 새로 생성된 인덱스의 index prefix이므로 삭제하는 것이 바람직하다.

db.getCollection(‘sales’).dropIndex({‘customer.gender’: 1, couponUsed: 1});
